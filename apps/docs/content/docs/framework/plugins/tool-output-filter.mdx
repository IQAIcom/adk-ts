The Tool Output Filter Plugin automatically detects and reduces oversized tool outputs using intelligent JQ filtering powered by LLM. It helps maintain optimal token usage by filtering verbose API responses, large datasets, and nested structures while preserving essential information.

## Key Features

- **Automatic Detection**: Monitors tool output size and key count to trigger filtering
- **Intelligent Filtering**: Uses LLM to generate context-aware JQ filters
- **Iterative Refinement**: Multiple filtering iterations for optimal size reduction
- **Configurable Thresholds**: Customizable size limits and target sizes
- **Tool-Specific Control**: Enable/disable filtering for specific tools
- **Security First**: Rejects dangerous JQ patterns (system commands, env access) and validates filter safety

## Quick Start

```typescript
import {
  InMemoryRunner,
  LlmAgent,
  LLMRegistry,
  ToolOutputFilterPlugin,
} from "@iqai/adk";

const agent = new LlmAgent({
  name: "data_assistant",
  description: "Analyzes large datasets",
  model: "gemini-1.5-pro",
});

const runner = new InMemoryRunner(agent, {
  appName: "my-app",
  plugins: [
    new ToolOutputFilterPlugin({
      filterModel: LLMRegistry.newLLM("gemini-1.5-flash"),
    }),
  ],
});
```

## Configuration

### FilterConfig

```typescript
interface FilterConfig {
  /** Size threshold in characters to trigger filtering (default: 8000) */
  sizeThreshold?: number;

  /** Key/field count threshold to trigger filtering (default: 50) */
  keyThreshold?: number;

  /** Target size after filtering (default: 4000) */
  targetSize?: number;

  /** Maximum filtering iterations (default: 3) */
  maxIterations?: number;

  /** Maximum depth for schema extraction (default: 4) */
  maxSchemaDepth?: number;

  /** Enable debug logging (default: false) */
  debug?: boolean;
}
```

### ToolOutputFilterPluginOptions

```typescript
interface ToolOutputFilterPluginOptions {
  /** Plugin name (default: "tool_output_filter_plugin") */
  name?: string;

  /** LLM model used to generate JQ filters (required) */
  filterModel: BaseLlm;

  /** Only filter outputs from these tools (undefined = filter all) */
  enabledTools?: string[];

  /** Skip filtering for these tools */
  disabledTools?: string[];

  /** Configuration for filtering behavior */
  config?: FilterConfig;
}
```

### Configuration Details

- **`filterModel`** _(required, `BaseLlm`)_
  - The LLM used to generate context-aware JQ filters
  - Use `LLMRegistry.newLLM("gemini-1.5-flash")` to create an instance from a model string
  - Use a fast/cheap model since filter generation is lightweight

- **`sizeThreshold`** _(default: 8000)_
  - Character count at which filtering is triggered
  - Outputs below this size pass through unchanged

- **`keyThreshold`** _(default: 50)_
  - Number of JSON keys at which filtering is triggered
  - Catches deeply nested structures even if character count is moderate

- **`targetSize`** _(default: 4000)_
  - The target output size after filtering
  - The plugin iterates until output is at or below this size (or max iterations reached)

- **`maxIterations`** _(default: 3)_
  - Maximum number of filter-and-check cycles
  - More iterations yield better compression but increase latency

- **`enabledTools` / `disabledTools`**
  - Whitelist or blacklist tools by name
  - If `enabledTools` is set, only those tools are filtered
  - If `disabledTools` is set, those tools are skipped

- **`debug`** _(default: false)_
  - Enables detailed logging: response size analysis, generated JQ filters, iteration progress, success/failure reasons

## How It Works

1. **Detection**: Monitors tool output size and key count against thresholds
2. **Analysis**: Extracts JSON schema and generates a preview of the data
3. **Filter Generation**: Uses LLM to create a context-aware JQ filter expression
4. **Application**: Applies JQ filter via spawned process (with 10-second timeout)
5. **Iteration**: Repeats until target size is reached or max iterations exceeded
6. **Metadata**: Adds filtering metadata to the result

When filtering is applied, the output includes metadata:

```json
{
  "_mcp_filtered": true,
  "_original_size": 15000,
  "_filtered_size": 4500,
  "_reduction_percent": 70.0,
  "_jq_filters": ["{id, name, summary}"],
  "_iterations": 2,
  "id": "123",
  "name": "Example Data",
  "summary": "Filtered content..."
}
```

If filtering fails for any reason (JQ error, timeout, dangerous pattern detected, filtered output larger than original), the plugin falls back to the original unfiltered output.

## Advanced Usage

### Custom Filter Model

Use `LLMRegistry.newLLM()` to create a `BaseLlm` instance from any registered model string:

```typescript
import { LLMRegistry, ToolOutputFilterPlugin } from "@iqai/adk";

const plugin = new ToolOutputFilterPlugin({
  filterModel: LLMRegistry.newLLM("gemini-1.5-flash"),
  config: { sizeThreshold: 10000, targetSize: 5000 },
});
```

### Performance Tuning

```typescript
// High-throughput: fewer filtering triggers, faster convergence
// LLMRegistry.newLLM() converts a model string to a BaseLlm instance
const highThroughput = new ToolOutputFilterPlugin({
  filterModel: LLMRegistry.newLLM("gemini-1.5-flash"),
  config: {
    sizeThreshold: 15000,
    targetSize: 8000,
    maxIterations: 2,
    maxSchemaDepth: 3,
  },
});

// Aggressive compression: lower thresholds, more iterations
const aggressive = new ToolOutputFilterPlugin({
  filterModel: LLMRegistry.newLLM("gemini-1.5-flash"),
  config: {
    sizeThreshold: 5000,
    targetSize: 2000,
    maxIterations: 5,
  },
});
```

### With Multiple Plugins

```typescript
import { InMemoryRunner, LLMRegistry } from "@iqai/adk";
import { ToolOutputFilterPlugin, ReflectAndRetryToolPlugin } from "@iqai/adk";

// Filter large outputs first, then handle errors
const runner = new InMemoryRunner(agent, {
  appName: "production-app",
  plugins: [
    new ToolOutputFilterPlugin({
      filterModel: LLMRegistry.newLLM("gemini-1.5-flash"),
    }),
    new ReflectAndRetryToolPlugin({
      maxRetries: 3,
    }),
  ],
});
```

## Best Practices

### When to Use

- **Large API Responses**: REST APIs returning verbose data
- **Database Queries**: Complex queries with nested relationships
- **File Processing**: Reading large JSON/XML files
- **External Integrations**: Third-party services with verbose responses

### When Not to Use

- **Small Outputs**: Outputs below size thresholds (unnecessary overhead)
- **Binary Data**: Non-JSON outputs (plugin only processes JSON)
- **Critical Data**: When every field is essential for downstream processing
- **Real-time Requirements**: When filtering latency is unacceptable

### Debugging

Enable debug mode to monitor filtering behavior:

```typescript
const plugin = new ToolOutputFilterPlugin({
  filterModel: LLMRegistry.newLLM("gemini-1.5-flash"),
  config: { debug: true },
});
// Logs: response size/key count, generated JQ filters, iteration progress
```

## Troubleshooting

**Plugin not filtering outputs:**

- Check if output exceeds size/key thresholds
- Verify tool is not in `disabledTools` list
- Ensure tool is in `enabledTools` list (if specified)

**Filtering taking too long:**

- Reduce `maxIterations`
- Increase `targetSize` for faster convergence
- Use a faster filter model

**Critical data being filtered out:**

- Increase `targetSize` to preserve more data
- Reduce `maxSchemaDepth` for more aggressive schema extraction
- Adjust `sizeThreshold` to trigger filtering less frequently

**JQ errors:**

- Check for malformed JSON in tool outputs
- Verify JQ is installed and accessible (`jq --version`)
- Enable `debug: true` for detailed error messages
