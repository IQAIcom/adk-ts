# OpenAPI Tools

OpenAPI tools enable integration with REST APIs based on OpenAPI specifications. ADK-TS will support automatic tool generation from OpenAPI specs in future releases.

<Callout type="info" title="Future Enhancement">
Automatic OpenAPI tool generation is planned for a future release. The current approach requires manual tool creation but provides full flexibility and control.
</Callout>

## Understanding OpenAPI Specifications

OpenAPI (formerly known as Swagger) is a specification format for REST APIs that describes available endpoints, operations, parameters, authentication requirements, and response formats. OpenAPI specifications can be in JSON or YAML format.

### Key OpenAPI Components

- **Paths**: Define available endpoints and HTTP methods
- **Parameters**: Specify path, query, header, and body parameters
- **Components**: Reusable schemas for requests and responses
- **Security Schemes**: Authentication requirements
- **Servers**: Base URLs for API requests

## Working with OpenAPI Specifications

When working with OpenAPI specifications, you'll need to:

1. **Parse the Specification**: Use an OpenAPI parser library to read and interpret the specification document.

2. **Analyze API Structure**:

   - Identify endpoints and available operations
   - Understand parameter requirements and data types
   - Map security requirements

3. **Generate TypeScript Types**: Use tools like `openapi-typescript` to generate type definitions from the OpenAPI schema:

```bash
# Install the tool
npm install --save-dev openapi-typescript

# Generate types from OpenAPI spec
npx openapi-typescript openapi.yaml --output schema.ts
```

## Current Implementation Approaches

Until native OpenAPI tool generation is available in ADK-TS, you can:

### Parse OpenAPI at Runtime

```typescript
import SwaggerParser from "@apidevtools/swagger-parser";
import { BaseTool } from "@iqai/adk";

async function createToolsFromOpenApi(specUrl: string) {
  try {
    // Parse and validate the OpenAPI spec
    const api = await SwaggerParser.validate(specUrl);

    // Create tools for each operation
    const tools = [];

    // Process API paths and operations
    for (const [path, pathItem] of Object.entries(api.paths)) {
      for (const [method, operation] of Object.entries(pathItem)) {
        if (method === "parameters" || method === "$ref") continue;

        // Create a tool for this operation
        const operationId = operation.operationId || `${method}-${path}`;

        // Create and add tool
        // (Actual implementation would be more complex)
        tools.push(new OpenAPIOperationTool(api, path, method, operation));
      }
    }

    return tools;
  } catch (error) {
    console.error("Error parsing OpenAPI spec:", error);
    return [];
  }
}

// Example OpenAPI operation tool
class OpenAPIOperationTool extends BaseTool {
  constructor(api, path, method, operation) {
    super({
      name: operation.operationId || `${method}-${path}`,
      description:
        operation.summary || operation.description || `${method} ${path}`,
    });

    this.api = api;
    this.path = path;
    this.method = method;
    this.operation = operation;
  }

  // Tool implementation would go here
}
```

## Best Practices for OpenAPI Integration

When building OpenAPI-based tools:

1. **Validate Specifications**: Ensure your OpenAPI document is valid before attempting to parse or generate code
2. **Handle Schema References**: Resolve `$ref` references properly when parsing specifications

3. **Type Safety**: Use generated TypeScript types to ensure type safety when interacting with APIs

4. **Version Management**: Provide clear version handling when the OpenAPI spec evolves

5. **Error Handling**: Handle specification parsing errors and API client errors gracefully
